# 大型工程分版本迭代规划

本 Skill 指导 Agent 与用户协作：先**钉死工程终点与产出物**，再按工程形态选择**纵向主路径**或**横向核心模块**拆解策略，输出**可对接 IDE 的版本迭代路径图谱**。

## 使用原则

- **先问清、再规划**：信息不足时主动追问，不臆造业务细节。
- **自动匹配场景**：根据用户描述的工程类型与交付形态，在「纵向路径型」与「横向广度型」中择一为主策略；边界模糊时说明假设并给出混合建议。
- **版本不堆叠**：每个版本有唯一重心，避免相邻版本大范围功能重叠；不遗漏「骨架 / 主路径 / 验证单元」等结构性环节。
- **默认技术选型**：在工程类型明确后给出**一套主推荐方案**（稳定、易落地），必要时用一行说明备选与切换条件。

---

## 一、工程基础定位（必须先完成）

在输出任何版本图谱前，引导用户明确以下四点；若任一项模糊，用**简短追问**补齐：

1. **最终目标**：从零到最终态，用户要得到的**完整产出物**是什么（可交付物、系统边界、成功判据）。
2. **最终形态**：面向谁、在什么环境运行、与哪些外部系统交互（若有）。
3. **核心价值**：解决什么问题；没有它时用户如何凑合。
4. **约束**：时间、合规、团队技能、必须沿用的技术栈或禁止项。

**产出（内部用）**：写一段「工程基线摘要」（3–8 句），作为后续所有版本目标的锚点。

---

## 二、技术结构选型建议（自动给出）

在「工程基线摘要」确定后，根据**工程大类**给出**主推荐**技术结构（命名具体技术栈或范式，避免「选一个框架」式开放式罗列）：

| 工程大类 | 主推荐关注点（示例方向，按实际裁剪） |
|----------|--------------------------------------|
| 纵向 Web / SaaS | 前后端边界、身份与租户模型、核心领域服务、可观测性、部署形态 |
| 流程 / 编排 / 自动化 | 状态机或工作流引擎、幂等与重试、人工审批节点、审计日志 |
| 移动端 + 后端 | 同步策略、离线、推送、API 版本与兼容 |
| 数据 / 分析平台 | 采集—存储—计算—服务分层、批流取舍、权限与血缘 |
| 横向组件库 / 设计系统 | 包管理、构建与产物格式、主题与 Token、文档与 playground |
| 实时 / 协作 / 游戏类 | 会话与匹配、同步协议、核心循环与 tick、资源加载 |

**输出要求**：用一个小节「**推荐技术结构**」列出：必选层次、主推荐栈（或仓库结构）、与工程类型的贴合理由；若有硬约束冲突，说明调整项。

---

## 三、工程类型判定（自动匹配拆解思路）

**纵向路径型**（优先用场景 1）：强调**端到端用户或系统旅程**，价值沿一条主链路兑现；例如云产品线某条业务闭环、流程化产品、从进入到退出的完整业务链路。

**横向广度型**（优先用场景 2）：强调**能力面扩展**，同一基座上持续增加模块或组件；例如 React/Vue 组件库、SDK、多模块平台、工具链集合。

**判定启发**（不必展示给用户，Agent 自行判断）：

- 若叙述以「用户走完一条流程」「调度—执行—结束」为主 → 纵向。
- 若以「先做底座再铺量」「组件 / 模块一个个加」为主 → 横向。
- 混合型：声明**主策略**（纵向或横向），次要维度在图谱中用「并行轨」或「后续版本挂载」说明。

---

## 四、分版本迭代核心逻辑

### 场景 1：纵向路径型工程

1. **靠前版本（V1 / V2）**  
   - 搭建**项目整体骨架**（目录、配置、最小可运行部署或本地运行方式）。  
   - 聚焦**核心纵向流程**，实现「**极简完整路径**」：从**初始调度** → **核心功能触发**（如进入主功能）→ **简单操作** → **正常退出**。  
   - **不**在此阶段加入与主路径无关的冗余功能。

2. **后续版本**  
   - 在已跑通的主路径上**逐步生长附加节点**：例如主路径前增加登录、全流程贯穿支付、调度中增加排队、登录后或核心功能内增加客服入口等。  
   - **每个版本聚焦 1–2 个附加节点**，保持迭代可控；标明与主路径的**挂载点**（在哪个步骤前/后插入）。

### 场景 2：横向广度型工程

1. **模块优先级**  
   - **先核心模块、后辅助模块**（例如先组件库本体，再文档站 / 演示平台）；**不**颠倒。

2. **V1：核心模块基础架构**  
   - 明确并落地：多语言方案、色彩与主题体系、设计 Token / 变量替换规则、安装与使用说明、可扩展的目录与构建配置等；形成**可扩展的基础框架**。

3. **V2：微小但完整的核心单元**  
   - 快速落地 **1–2 个**基础且端到端可用的单元（如基础组件），用于**验证基础架构可行性**（发布、引用、主题、文档其中至少一条链路透）。

4. **后续版本**  
   - 在基础架构与初始单元之上**横向扩展**核心模块（更多组件 / 能力）；**同步**推进辅助模块（文档、示例、脚手架），避免长期「只有库没有落地证明」。

## 六、质量自检（输出前在内心完成）

- [ ] 每个版本是否有**唯一**重心且与相邻版本**少重叠**？
- [ ] 是否覆盖「骨架 → 最小闭环 / 最小单元 → 扩展」中的应有环节（按场景）？
- [ ] **依赖**是否只指向前序版本，无循环依赖？
- [ ] 「显式不包含」是否写清，避免范围蔓延？
- [ ] 用户若未给工程类型，是否已通过追问或合理假设并在图谱中写明？

## 七、与用户交互的简短话术示例

- 信息不足：「为把 V1 缩到可两周内落地，需要确认：最终产出物是面向内部还是外部？必须上线的最小流程是哪三步？」
- 类型模糊：「从描述看更像纵向产品（一条主流程）；若你更关心先做齐 20 个 API 再拼流程，我们可以按横向平台微调——你更认同哪一种？」

按本 Skill 执行时，**以用户诉求中的两种场景逻辑为准**，不在此之外发明第三套主逻辑；混合工程仅在「场景判定」中说明主次与衔接方式即可。
