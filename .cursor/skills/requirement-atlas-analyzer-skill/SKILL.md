# Requirement Atlas Analyzer（需求图谱分析）

## 目的

在用户已描述**某一具体新需求**的前提下，分析该需求与**当前仓库已有代码**之间的关联，把**常相关 / 强相关**的落点、行为差异、主逻辑变动与连锁影响，按 **主路径 → 分支逻辑 → 边界条件** 整理成一份 **Markdown 参考文档**，供后续实现阶段快速加载上下文。

本 skill **不做**完整需求规格说明书、**不替代**逐文件精读与测试设计；产出是**浓缩的关系与影响视图**。

**范围边界**：只分析**本条需求**与仓库的关联与影响；默认输出为 `docs/requirements/` 下的独立分析文档，**不要**把本文档与其它文档合并为同一文件，除非用户明确要求。

## 输入前提（执行前）

1. 用户在本轮对话中给出**可分析的需求描述**（可粘贴 PRD/工单/列表）；若过于模糊，应先列出待澄清问题，再仍生成文档但将不确定项单独标注。
2. 需求范围应相对单一；若用户一次抛出多个大需求，应拆成多份文档或让用户指定优先级最高的一条。

## 默认产物路径

在仓库中创建目录（若不存在）并写入：

`docs/requirements/<slug>-requirement-atlas.md`

- `<slug>`：由需求标题或关键词转成的短 kebab-case（如 `user-profile-export`）；若无法概括，用 `req-YYYYMMDD`。
- 同一需求迭代更新时，**覆盖**同一文件并在文内「修订记录」追加一行日期与摘要，或用户指定版本后缀。

## 生成文档的硬性要求：文首与文末声明

生成的 MD **必须在最开头**（在 YAML 元数据之后、正文第一节之前）包含一块 **「文档性质与局限」**，并在**文档最末尾**再次用独立小节重复核心警告（允许与文首略同义压缩，但**不得省略**下列要点）：

1. **本文档仅供实现阶段参考**，不是权威需求规格；**未**保证覆盖需求全部细节，后续仍可能补充或修正需求。
2. 文中的关联与影响关系基于**生成时刻**的仓库与描述整理，**知识 / 相关性图谱可能不全面**；已写内容力求有据，但**可能准确却不完整**。
3. 在逐步实现过程中，仓库会持续变化，**本文档可能逐步与真实代码不一致**；使用时应**谨慎对照**当前仓库，以代码与测试为准，必要时回到本文件仅作「当时如何理解关系」的线索。

鼓励在文首使用 **YAML frontmatter**（或 HTML 注释块）承载机器可读元数据，便于后续模型判断文档类型与新鲜度，例如：

```yaml
---
doc_type: requirement-atlas-analyzer-skill
generated_at: "2026-05-04"
requirement_slug: "<slug>"
related_paths_glob: "src/features/foo, apps/web/api"
staleness_note: "Snapshot; repo may diverge during implementation."
confidence: "high|mixed|low"
---
```

`confidence` 可概括「需求描述清晰度 + 代码证据充分度」。

## 正文结构与信息顺序

按下列章节组织（可删「不适用」节，但**禁止**打乱「从主到次」的叙述逻辑）：

1. **需求摘要（引用级）** — 用户原意的一句话概括 + 成功判据（若用户已给）。
2. **涉及范围总览** — 表格：`路径或模块 | 与需求关系（读/写/配置/类型） | 证据（函数/路由/关键符号）`。
3. **当前行为（As-Is）** — 与需求相关的现有表现，只写有代码依据的部分。
4. **目标行为（To-Be）** — 实现完成后预期表现；区分「已明确」与「推断」。
5. **主功能逻辑变动** — 主路径上会改什么（数据流、状态机、API 契约、持久化等）。
6. **分支与影响** — 次级分支：哪些旧分支可能**删除或废弃**；新逻辑可能引入哪些**小分支**；对其它功能/模块的**耦合影响**。
7. **边界条件与风险** — 空值、并发、权限、兼容版本、迁移、性能、安全面；列出**待补测试**点。
8. **待确认清单** — 需求侧与代码侧仍缺的信息。
9. **修订记录** — 日期 + 一行变更说明（首次写「初稿」）。

每一节内，叙述顺序遵守：**主路径 → 分支逻辑 → 边界条件**。

## 篇幅与证据

- **浓缩**：优先表格与列表；避免大段粘贴源码，必要时用「文件:行号或符号名」指代。
- **有据则写、无据则标「待确认」**，禁止为凑全而编造文件路径。
- 安全与合规：若涉及敏感操作，单列「安全与权限影响」子段，仍保持索引级，不写 exploit。

## 执行建议流程（给代理）

1. 解析用户给出的需求，列出 3～7 个**关键词 / 能力点**。
2. 从仓库根部的说明与清单类文件（如 `README*`、清单与配置文件）入手建立粗粒度方位，再结合 `grep`/语义搜索定位与需求相关的候选路径。
3. 打开**最少必要**的关键文件验证调用关系，更新「涉及范围总览」表。
4. 按模板写出完整 MD，**文首文末声明**与 **frontmatter** 不可漏。
5. 告知用户文件路径；提醒实现过程中若结构大变，应**重跑本 skill** 或手改该文档。

## 禁止事项

- 不要把本文档写成「最终技术方案定稿」或大量实现代码。
- 不要在本 skill 产出中夹带与**当前用户所述需求无关**的全仓概论；笔墨集中在与本需求相关的路径、行为与影响上。
